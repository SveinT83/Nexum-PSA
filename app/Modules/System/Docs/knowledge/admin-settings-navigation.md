Admin settings are grouped from the Admin hub and the shared Admin side menu.

The Admin hub should expose settings surfaces that are implemented and beta-ready. It must not advertise unfinished settings pages.

## Main Groups

The Admin hub currently groups settings into:

- Commercial.
- Economy.
- Email.
- Sales.
- Clients.
- Calendar.
- Storage.
- Tickets.
- Templates.
- Users.
- System.
- Integrations.

## Discoverability Rule

If a production-ready settings route exists, it should be reachable from:

- The Admin landing page.
- The Admin side menu when that area is active.

If a settings route exists but is not beta-ready, it should stay hidden until the underlying behavior is implemented and tested.

## Current Exposed Settings

Existing settings surfaces include:

- Calendar settings.
- Notification channels.
- Nextcloud connections.
- N-able RMM.
- Tactical RMM.
- BookStack.
- API management.
- AI settings.
- User roles.
- User permissions.
- Two-factor auth.
- Ticket technician assignment settings.
- Ticket assignment rules.

## Future Settings Work

Some active modules still need settings ownership decisions before they get admin routes.

Examples:

- Assets.
- Contacts.
- Knowledge.
- Risk.
- Tasks.
- Warroom.
- Reports.

Those settings should be designed through the missing settings ownership RFC before implementation.
